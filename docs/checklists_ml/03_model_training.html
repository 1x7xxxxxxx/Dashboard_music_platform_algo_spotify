§ 4.1 — Algorithm Selection

Algorithm selection by task — overview

Six task families, each with a baseline / intermediate / advanced / AutoML tier. Always start with the baseline before reaching for XGBoost or a transformer.

  • Classification — discrete label prediction
  • Regression — continuous value prediction
  • Anomaly detection — flag rare deviations
  • Time series — forecast or classify ordered samples
  • Clustering — discover unknown groups
  • Reinforcement learning — sequential decision-making under reward signal
§ 4.1 — Classification

Classification — algorithm tier list

Baseline · always run first
DummyClassifier
strategy='most_frequent' — sets the floor any real model must beat. If your XGBoost barely beats this, you have a data leakage or a class-balance problem.
Intermediate · interpretable
LogReg · NB · SVM · KNN · Decision Tree · Random Forest
Strong on small-to-medium tabular. Easy to inspect coefficients / feature importance. Use when audit / regulator requires explainability.
Advanced · production default
XGBoost · LightGBM · CatBoost
Boosted trees — typically the best non-DL choice on tabular. Native handling of NaNs, categorical, class weights, early stopping.
AutoML · strong baseline
TPOT
Genetic algorithms evolve full pipelines (preprocessing + model + HPO). Slow but produces human-readable code. Use as benchmark, not as final model.
§ 4.1 — Regression

Regression — algorithm tier list

Baseline
DummyRegressor
Predicts the mean (or median) of the training target. Beating this by less than 5 % means the model is not learning anything meaningful.
Intermediate
LinearRegression · SVR · KNN · Decision Tree
Linear regression first to read coefficients; SVR for non-linear with a small dataset; KNN when local structure dominates.
Advanced
XGBoost · LightGBM
Same boosted-tree story as classification. Use objective='reg:squarederror' or quantile losses for prediction intervals.
AutoML
TPOT
Same caveats — strong baseline, not a black-box for production without explainability gate.
§ 4.1 — Anomaly Detection

Anomaly detection — algorithm tier list

Baseline
z-score threshold
Flag samples with |x − μ| / σ > 3. Trivial to implement, surprisingly hard to beat on Gaussian-distributed signals.
Intermediate
Isolation Forest
Builds random partition trees; anomalies isolate faster (shorter path). Linear time, no distance computation, robust to high dimensions.
Advanced
PCA-GMM · Autoencoder
Learn the manifold of healthy data; flag samples with high reconstruction error or low GMM likelihood. Required when modes are non-Gaussian.
AutoML
— (not common)
Few mature AutoML libraries cover anomaly detection well. PyOD provides a unified API across 40+ algorithms — closest to AutoML.
§ 4.1 — Time Series

Time series — algorithm tier list

Baseline
Last-value baseline
y_pred[t+1] = y[t]. The bar a real forecasting model must clear. Many "deep TS" papers don't beat this on industrial signals.
Intermediate
ARIMA
Statistical workhorse. Use auto-ARIMA (pmdarima.auto_arima) to skip the manual (p, d, q) search. Requires stationary input — apply differencing if needed.
Advanced
LSTM · RNN · TCN
TCN is preferred over LSTM for sequences > 200 steps: parallelizable training, stable gradients, larger receptive field.
AutoML
— (use specialized libs)
Use Nixtla StatsForecast / NeuralForecast or Darts — covered in §7.6.
§ 4.1 — Clustering

Clustering — algorithm tier list

Baseline
Random assign
Sanity check: any clustering you propose must produce groups with silhouette > random assignment. Surprisingly often violated.
Intermediate
K-Means
Convex, fast, scales to millions of points. Requires k upfront — find via elbow on inertia or silhouette analysis.
Advanced
DBSCAN · Hierarchical
DBSCAN handles noise (−1 label) and arbitrary shapes — no k required, but two parameters (eps, min_samples) to tune. Hierarchical gives a dendrogram you can cut at any level.
AutoML
— (limited)
No mainstream AutoML for clustering. Closest: HDBSCAN with default parameters often works without tuning.
§ 4.1 — Reinforcement Learning

Reinforcement learning — algorithm tier list

Baseline
Random policy
Always log the average reward of a uniformly-random policy. Many RL setups have a near-optimal random baseline that exposes a broken reward shape.
Intermediate
Q-Learning
Tabular value-based method. Use only when state and action spaces are small enough to materialize the Q-table.
Advanced
PPO · SAC (Stable-Baselines3)
PPO = on-policy, conservative updates, robust to hyperparameters. SAC = off-policy, sample-efficient, max-entropy formulation. Default to PPO unless sample efficiency is critical.
AutoML
— (research only)
RL AutoML (Auto-RL) is academic. Expect to hand-tune RL hyperparameters for the foreseeable future.
§ 4.1 — Recap

Algorithm-selection — take-aways

  • Always run the baseline first. If you cannot beat it by ≥ 5 %, the model is not learning the signal.
  • Reach for XGBoost / LightGBM by default on tabular — they outperform deep learning on most industrial datasets.
  • Boosted trees handle missing values, categorical features, class imbalance natively. Do not pre-impute or one-hot prematurely.
  • Do not use a transformer on a 5 k-row dataset. It will memorize, not generalize.
  • Do not ship AutoML to production without an explainability gate (§5.3) and an enumerable ensemble.
  • Document your algorithm choice in an ADR with the discarded alternatives and the discard reason.

§ 4.2 — Item 4.1

Baseline first DO

Always fit a trivial model before any real training. The baseline gives you the floor — a number you must beat by a meaningful margin.

  • Classification: DummyClassifier(strategy='most_frequent') or 'stratified'
  • Regression: DummyRegressor(strategy='mean') or 'median'
  • Time series: y_pred[t+1] = y[t] (last-value)
  • Anomaly: z-score threshold or rolling-quantile rule

Decision rule: if your real model beats the baseline by < 5 % on the metric of interest, treat it as a red flag — likely a data leakage upstream or an over-aggressive train-test split.

§ 4.2 — Item 4.2

Cross-validation — pick the right splitter METHOD

k-fold (standard)
cross_val_score(model, X, y, cv=5). Good default for IID tabular data with enough samples per class.
StratifiedKFold
Preserves class ratio per fold. Mandatory for any imbalanced classification — otherwise some folds may have 0 positive samples.
TimeSeriesSplit
Temporal order preserved. Train always before val. NEVER use random k-fold on time-ordered data — it leaks the future into training.
GroupKFold
Samples belong to natural groups (per machine, per patient). Same group cannot appear in both train and val — otherwise leakage of group-specific noise.
Nested CV
Outer loop = model selection · inner loop = HPO. Gives an unbiased generalization estimate. Costly — use for the final reported metric only.
§ 4.2 — Item 4.3

Stratified CV — verify class presence DO

Even with StratifiedKFold, an extreme imbalance (1 positive per 10 000 rows) can land a fold with zero positives. Verify explicitly before training.

from sklearn.model_selection import StratifiedKFold
import numpy as np

skf = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
for fold, (tr, va) in enumerate(skf.split(X, y)):
    assert len(np.unique(y[tr])) > 1, f"fold {fold} train has 1 class"
    assert len(np.unique(y[va])) > 1, f"fold {fold} val has 1 class"

§ 4.2 — Item 4.4

Hyperparameter search — escalation ladder METHOD

Do not jump straight to Optuna. Climb the ladder — each step costs 10× more compute than the previous one.

  • 1. GridSearchCV — exhaustive search of a small grid (≤ 100 combinations). Best when you know the right ballpark already.
  • 2. RandomizedSearchCV — random sampling, n_iter=50 typical. Better coverage of large search spaces, faster than grid.
  • 3. Optuna (Bayesian TPE) — learns from past trials, suggests smarter values, supports pruning bad trials early.
  • 4. Hyperopt — alternative Bayesian optimizer, similar API.
  • 5. Hyperband / ASHA — multi-fidelity: launches many configs with small budgets, promotes survivors. Best for deep learning where epochs are expensive.

Whatever you choose: log every trial in MLflow with the seed, the score, the config, and the wall-clock time.

§ 4.2 — Item 4.5

Early stopping — boosted trees DO

XGBoost / LightGBM / CatBoost grow trees iteratively. early_stopping_rounds halts training as soon as the validation metric stops improving — preventing the model from memorizing training noise.

import xgboost as xgb

model = xgb.XGBClassifier(
    n_estimators=2000,           # upper bound — early stop will cut it
    learning_rate=0.05,
    max_depth=6,
    early_stopping_rounds=50,    # stop if val metric flat for 50 rounds
    eval_metric='aucpr',
)
model.fit(
    X_train, y_train,
    eval_set=[(X_val, y_val)],
    verbose=False,
)
print(f"stopped at iteration {model.best_iteration}")
§ 4.2 — Item 4.6

Overfitting diagnosis — learning curves WATCH

Plot train and val metric vs training progress. The shape tells you what to do next.

Pattern: train ≫ val
Overfit
Add regularization (L2, dropout, smaller model), add data, augment, lower LR, early stop sooner.
Pattern: train flat, low
Underfit
See item 4.13. Increase capacity, remove regularization, add features.
Pattern: train + val converging
Sweet spot
Both metrics improving, gap stable < 10 %. Continue training, then early-stop on val plateau.

Always log the curves to MLflow as artifacts (PNG of matplotlib) — debugging a regression in 6 months is impossible without them.

§ 4.2 — Item 4.7

Threshold optimization — never ship 0.5 DO

A trained classifier outputs probabilities. The default 0.5 cutoff is rarely optimal — pick the threshold that maximizes the metric you actually care about, on the validation set.

from sklearn.metrics import precision_recall_curve, f1_score
import numpy as np

probs = model.predict_proba(X_val)[:, 1]
prec, rec, thresh = precision_recall_curve(y_val, probs)
f1 = 2 * prec * rec / (prec + rec + 1e-9)
best_idx = np.argmax(f1)
best_threshold = thresh[best_idx]

print(f"best F1 = {f1[best_idx]:.3f} at threshold = {best_threshold:.3f}")
# Persist the threshold next to the model — required for inference

Store best_threshold in the model bundle (e.g. model_card.json) — inference code MUST read it instead of hard-coding 0.5.

§ 4.2 — Item 4.8

Cost-sensitive training — handle imbalance METHOD

Three weapons against class imbalance, ranked by simplicity. Try in order; SMOTE is a last resort because it inflates synthetic positives.

  • 1. scale_pos_weight=N_neg / N_pos — XGBoost / LightGBM. The simplest, no data manipulation.
  • 2. class_weight='balanced' — sklearn classifiers (LogReg, SVM, RandomForest, etc.). Reweights the loss.
  • 3. sample_weight=... — fit-time array. Use when individual samples have known importance (recency, business value).

Anti-pattern: applying class weights AND oversampling AND custom loss simultaneously. The effects compound unpredictably — pick one strategy and document the choice.

§ 4.2 — Item 4.9

Model stacking — proceed with caution WATCH

Stacking trains a meta-model on the out-of-fold predictions of base models. It often wins Kaggle competitions; in production it adds latency, complexity, and a new failure mode.

  • Use StackingClassifier from sklearn — it handles out-of-fold prediction generation correctly.
  • For time series: meta-model MUST not see future data — use TimeSeriesSplit in the stacking CV, otherwise you leak the future into the meta-features.
  • Inference latency = sum of all base models + meta-model. Verify the p95 against your SLO before shipping.
  • Each base model is a separate retrain target — set up the retraining DAG accordingly.
  • Skip stacking unless a single best model has plateaued AND the marginal gain (≥ 2 % on the target metric) justifies the operational cost.
§ 4.2 — Item 4.10

Neural Architecture Search — heavy artillery EXPENSIVE

NAS automates the design of the network architecture itself (depth, width, connections). It is famously compute-hungry — DARTS, ENAS, and similar methods consume hundreds of GPU-hours.

  • Use AutoKeras for vision/text — abstracts NAS behind a simple .fit() call.
  • Use nni (Microsoft) or syne-tune for full NAS pipelines.
  • Document a strict compute budget in the ADR — NAS without a budget runs forever.
  • For tabular: NAS is almost never worth it. Use AutoML libraries like AutoGluon or FLAML instead.
  • Default for industrial: do not use NAS. Hand-design the network from a known reference, fine-tune via Optuna / Hyperband.
§ 4.2 — Item 4.11

Reproducibility — fix every seed MANDATORY

A model that produces different metrics on re-run is not auditable. Fix every seed BEFORE any split, fit, or training call.

import os, random
import numpy as np
import torch
import tensorflow as tf

SEED = 42
os.environ["PYTHONHASHSEED"] = str(SEED)
random.seed(SEED)
np.random.seed(SEED)
torch.manual_seed(SEED)
torch.cuda.manual_seed_all(SEED)
torch.backends.cudnn.deterministic = True
torch.backends.cudnn.benchmark = False
tf.random.set_seed(SEED)

mlflow.log_param("seed", SEED)   # MLflow audit trail

Acceptance test: rerun the training pipeline from the same git SHA + same data version; metrics must match within float tolerance.

§ 4.2 — Item 4.12 (NEW)

LR adjustment strategy — the trio DO

A learning-rate scheduler alone is not enough. Pair it with early stopping AND best-weights restore — the three together form one coherent training loop.

  • (a) Scheduler — lowers LR when val_loss plateaus: ReduceLROnPlateau, CosineAnnealingLR, OneCycleLR.
  • (b) Early stopping — halts training after N epochs without improvement on val_loss.
  • (c) ModelCheckpoint(save_best_only=True) — keeps the best weights ever seen, restored on stop.
  • Boosted trees variant: early_stopping_rounds + a learning_rate callback (e.g. shrink LR every 100 rounds without improvement).
  • KMeans / GMM: no LR but pin tol=1e-4 and max_iter=300 as the analog convergence criterion.
  • Always log the scheduler config + the final LR + the stop epoch to MLflow.
§ 4.2 — Item 4.12 — Code

LR scheduler + early stopping — PyTorch reference

import torch
from torch.optim.lr_scheduler import ReduceLROnPlateau

optim = torch.optim.Adam(model.parameters(), lr=1e-3)
scheduler = ReduceLROnPlateau(optim, mode='min', factor=0.5, patience=5)

best_loss, no_improve, best_state = float('inf'), 0, None
PATIENCE = 15

for epoch in range(MAX_EPOCHS):
    train_loss = train_one_epoch(model, train_loader, optim)
    val_loss   = evaluate(model, val_loader)
    scheduler.step(val_loss)             # LR drops on plateau

    if val_loss < best_loss:
        best_loss, no_improve = val_loss, 0
        best_state = {k: v.clone() for k, v in model.state_dict().items()}
    else:
        no_improve += 1
        if no_improve >= PATIENCE: break     # early stop

model.load_state_dict(best_state)        # restore best weights
§ 4.2 — Item 4.13 (NEW)

Underfitting diagnosis — the symmetric check CRITICAL

Item 4.6 catches overfitting (train ≫ val). Item 4.13 catches underfitting (both train and val high, gap small). Without 4.13, a poorly-fit model looks "consistent" and ships.

  • After fit, compute BOTH train_loss and test_loss.
  • gap = test_loss − train_loss
  • Log the triple (train_loss, test_loss, gap_pct) as MLflow params — required for audit.
§ 4.2 — Item 4.13 — Regimes

Three regimes — read the gap

train_loss high · gap < 5 %
Underfit
Action: increase capacity (tree depth, hidden units, more features), or remove regularisation. Never accept high train_loss as "good fit".
train_loss low · gap < 10 %
Sweet spot
Action: ship. Continue with calibration (§5.5) and threshold selection (§4.7).
gap > 20 %
Overfit
Action: regularise (cf §4.6) — dropout, L2, fewer features, more data, augmentation.

NEVER report only test_loss. Without the train baseline, underfitting is invisible — and a model that under-fits will keep under-fitting on production data.

§ 4.13 — Illustration

Bias-variance tradeoff — the U-curve

As model complexity grows, bias drops (model can fit more), variance grows (model overfits noise). Total error has a minimum — the sweet spot.

§ 4.2 — Item 4.14 (NEW)

Random restart / multi-init — escape local minima DO

For non-convex algorithms (NN, KMeans, GMM, autoencoders, t-SNE, PPO), a single seed converges to a single local minimum. Train multiple times with different seeds, keep the best.

  • (a) sklearn natively: KMeans(n_init=10), GaussianMixture(n_init=5).
  • (b) PyTorch / TF: train N=5 models with seeds [42, 123, 456, 789, 1010], keep the one with best val_loss.
  • (c) Logging: every seed gets its own MLflow run_id; promote only the best.
  • Different from item 4.11 reproducibility — that fixes ONE seed for replay; this EXPLORES many seeds.
  • Health check: std(val_loss) across N runs < 5 % of mean = stable. > 10 % = revisit LR or regularization.
§ 4.2 — Item 4.14 — Code

Multi-init — orchestrate N runs in MLflow

import mlflow, numpy as np

SEEDS = [42, 123, 456, 789, 1010]
results = []

with mlflow.start_run(run_name="multi_init_parent") as parent:
    for seed in SEEDS:
        with mlflow.start_run(run_name=f"seed_{seed}", nested=True):
            set_all_seeds(seed)              # cf item 4.11
            model = build_model()
            val_loss = train_and_eval(model)
            mlflow.log_param("seed", seed)
            mlflow.log_metric("val_loss", val_loss)
            results.append((seed, val_loss, model))

    seeds, losses, models = zip(*results)
    best_i = int(np.argmin(losses))
    mlflow.log_metric("best_val_loss", losses[best_i])
    mlflow.log_metric("multi_init_std_pct", 100 * np.std(losses) / np.mean(losses))
    mlflow.log_param("best_seed", seeds[best_i])
    mlflow.pytorch.log_model(models[best_i], "best_model")

§ 4.2 — Recap

Training checklist — the 14 in one breath

  • Always run a baseline; fail loudly if real model does not beat it by ≥ 5 %.
  • Pick the right CV splitter (StratifiedKFold, TimeSeriesSplit, GroupKFold) — defaults bite hard.
  • Climb the HPO ladder: Grid → Random → Optuna → Hyperband. Do not skip levels.
  • Pair LR scheduler + early stopping + best-weights restore — the trio (4.12).
  • Compute and log both train_loss and test_loss to detect underfitting (4.13).
  • Multi-init for non-convex algos; promote the best of N runs (4.14).
  • Fix every seed, log every config, persist every threshold.

§ 4.3 — HPO companion

Why pick which HPO method

GridSearchCV
Exhaustive Cartesian product. Use only when search space ≤ 100 combinations and trial cost is low. Reproducible by definition.
RandomizedSearchCV
Random sampling, fixed n_iter. Often outperforms grid because most hyperparameters don't matter — random covers the few that do, faster.
Optuna (TPE)
Tree-structured Parzen Estimator: builds two distributions (good vs bad trials), suggests new points where good is high. Prunes underperformers early via MedianPruner.
Hyperband / ASHA
Multi-fidelity: launches many short trials, doubles budget for survivors. Async ASHA = best GPU usage on a cluster — workers never wait for the slowest trial.
§ 4.3 — Optuna minimal

Optuna — 8-line reference

import optuna
from sklearn.metrics import f1_score

def objective(trial):
    params = {
        "max_depth":    trial.suggest_int("max_depth", 3, 10),
        "learning_rate": trial.suggest_float("lr", 1e-3, 0.3, log=True),
        "subsample":    trial.suggest_float("subsample", 0.5, 1.0),
    }
    model = xgb.XGBClassifier(**params, n_estimators=500,
                              early_stopping_rounds=30)
    model.fit(X_train, y_train, eval_set=[(X_val, y_val)], verbose=False)
    return f1_score(y_val, model.predict(X_val))

study = optuna.create_study(direction="maximize",
                            pruner=optuna.pruners.MedianPruner())
study.optimize(objective, n_trials=100, timeout=3600)
print(study.best_params, study.best_value)

§ 4.4 — Workflow

AutoML workflow — recommended sequence

  • 1. Run AutoGluon or FLAML with a fixed time budget (e.g. 1 hour) on the SAME train split as your custom model.
  • 2. Compare hold-out metrics. If AutoML wins by > 2 % F1 and you cannot explain why, treat as red flag: likely data leakage or a preprocessing bug in your pipeline.
  • 3. Use the AutoML leaderboard's feature importance to seed your custom model — pick the top-15 features it surfaced.
  • 4. Register the AutoML model in MLflow as baseline_automl EVEN if you ship your custom — it becomes your rollback target.
  • If you ship AutoML directly, the gates on the next slide are non-negotiable.
§ 4.4 — Production gates

Gates before shipping AutoML to production MANDATORY

  • Model card includes the AutoML library version + seed + time budget used.
  • Ensemble members are enumerable (you can list each base learner) — not an opaque blob.
  • Inference latency p95 measured against your SLO. AutoGluon stacks can be slow at predict time.
  • Retraining script is re-runnable end-to-end with the exact same config — version it in git.
  • Explainability: SHAP / feature importance exported to the model card (cf §5.3).
  • Without all of the above, the model is not auditable — do not deploy.
§ 4.4 — Anti-patterns

AutoML — three classic mistakes

  • Time budget too small. Running AutoML with time_limit=60 and declaring victory — most libraries need ≥ 30 min to converge.
  • Letting AutoML choose features from the full table, including leakage columns like repair_date or resolution_status. Always pre-filter.
  • Comparing apples to oranges: AutoML on one seed vs your custom model on a different seed. Use the same seed and same data split for any comparison to be meaningful.
  • Trusting the leaderboard score without a held-out test set. AutoML leaderboards are often computed on internal CV — independent test set still required.

§ 5.1 — Classification metrics

Classification metrics — there is no "best" one

The right metric depends on the cost of false positives vs false negatives in your business. Picking the wrong one ships an "accurate" model that destroys value.

  • Accuracy — only safe when classes are balanced.
  • Precision — when false alarms are expensive (each costs an unnecessary maintenance intervention).
  • Recall — when missed positives are catastrophic (safety-critical failures).
  • F1 — when both errors cost roughly the same.
  • ROC-AUC — model's ranking ability across all thresholds.
  • PR-AUC — better than ROC for extreme imbalance (rare-event detection).
§ 5.1 — Accuracy & Precision

Accuracy & Precision

Use only on balanced classes
Accuracy = (TP + TN) / N
Naive metric. On 1 % positive imbalance, predicting "always negative" yields 99 % accuracy. Anti-pattern in industrial fault detection.
Minimize false alarms
Precision = TP / (TP + FP)
"Of all cases I flagged as fault, how many were real?" Use when each false alarm triggers a costly maintenance dispatch.
§ 5.1 — Recall & F1

Recall & F1

Minimize missed faults
Recall = TP / (TP + FN)
"Of all real faults, how many did I catch?" Use in safety-critical contexts where missing a fault has high consequences (downtime, accidents).
Balanced trade-off
F1 = 2·P·R / (P + R)
Harmonic mean of precision and recall. Default when neither error dominates. Robust on imbalanced data — strictly better than accuracy.

ROC-AUC & PR-AUC

Threshold-independent
ROC-AUC
Area under the ROC curve (TPR vs FPR). Measures the model's ability to rank positives above negatives. 0.5 = random; 1.0 = perfect. Misleading when classes are heavily imbalanced.
Imbalance-friendly

§ 5.2 — Regression metrics

Regression metrics — pick by error scale & outlier sensitivity

  • R² / Adj-R² — fraction of variance explained. Easy to interpret, comparable across datasets.
  • Explained variance — like R² but ignores systematic bias. Use to separate variance error from offset error.
  • MAE — mean absolute error. Robust to outliers, in same unit as target.
  • MSE / RMSE — penalizes large errors quadratically. Use when big mistakes are disproportionately costly.
  • MSLE — log-scale error. Use for exponential-growth targets (population, revenue).
§ 5.2 — R² & Explained Variance

R² & Explained variance

Goodness of fit
R² / Adjusted R²
Fraction of variance in the target explained by the model. Adj-R² penalizes adding spurious features. Easy comparison across datasets — but a 0.95 R² on a noisy industrial signal can still be useless if the residuals are autocorrelated.
Bias separation
Explained variance
1 − Var(y − ŷ) / Var(y). Unlike R², does NOT penalize a systematic constant offset (bias). Use when you want to separate bias from variance error sources. explained_variance_score() in sklearn.
§ 5.2 — MAE & MSE/RMSE

MAE & MSE / RMSE

Robust, interpretable
MAE = mean(|y − ŷ|)
Average absolute error, in the same unit as the target. Robust to outliers — a single huge error doesn't dominate. Default for industrial regression.
Penalize large errors
MSE / RMSE
mean((y − ŷ)²). Squaring penalizes outliers heavily. RMSE = √MSE puts the error back into the target unit. Use when a 10× error is more than 10× as bad as a 1× error.
§ 5.2 — MSLE & plots

MSLE + mandatory regression plots

Exponential targets
MSLE
mean((log(1+y) − log(1+ŷ))²). Use when target values span orders of magnitude (population, revenue, sensor counts). Treats relative error, not absolute.
Required diagnostic plots
Plots
Residual plot · Breusch-Pagan heteroscedasticity · Q-Q plot for normality · prediction-error scatter · MAE vs epoch · LassoCV alpha sweep (lasso.mse_path_).

§ 5.5 — Probability Calibration

Calibration — does predict_proba(x) == 0.9 mean P(positive) ≈ 0.9?

A model can have great ROC-AUC but badly calibrated probabilities — predicting 0.9 when the true probability is 0.6. If your downstream uses the probability VALUE (risk score, expected cost), calibration is mandatory.

  • Random forests under-confident; SVMs / boosted trees over-confident — both common.
  • Decision rule: if you only need the ranking, calibration is optional. If the value matters, calibration is mandatory.
  • Wrap any sklearn estimator with CalibratedClassifierCV after training.
  • Plot the calibration curve — perfect calibration is a diagonal in the [0, 1] square.
§ 5.5 — Methods

Calibration methods — three options

Logistic on raw scores
Platt Scaling
Fits a logistic regression on top of model output. Cheap, works on small datasets. Originally designed for SVMs.
Non-parametric monotone
Isotonic Regression
Fits a monotone step function. More flexible than Platt; needs more data to avoid overfitting (≥ 1k samples).
Wraps any sklearn model
CalibratedClassifierCV
Cross-validated wrapper that produces a calibrated copy of any base estimator. Default to method='isotonic' on ≥ 1k val samples.

§ 5.6 — Causal Inference

Causal

Correlation ≠ causation

A standard ML model answers "what will happen?" (prediction). Causal inference answers "what would happen if I did X?" (intervention). Most industrial questions are secretly causal — and a classifier cannot answer them.

  • "Does the new PID tuning reduce wear?" — causal.
  • "Did the coolant upgrade reduce fault rate?" — causal.
  • "Which maintenance action lowers MTBF?" — causal.
  • Predictive models exploit correlation; deploying a decision based on a non-causal correlation is how you get production surprises.
§ 5.6.a — Vocabulary (1/2)

Core vocabulary — first half

Treatment T
The intervention whose effect we want to measure (1 = applied, 0 = not).
Outcome Y
The metric we care about — wear index, downtime, fault rate.
Confounder X
A variable that influences both T and Y. If uncontrolled, biases the estimate.
Mediator M
T → M → Y; part of the causal path. Do NOT control for it when estimating total effect.
Collider
T → C ← Y; controlling for C creates a SPURIOUS association.

§ 5.6.a — Vocabulary (2/2)

Core vocabulary — second half

ATE — Average Treatment Effect
E[Y(1) − Y(0)] across the whole population. The headline number.
ATT — ATE on Treated
E[Y(1) − Y(0) | T=1]. Effect on those who actually received the treatment — typically asked in marketing.
CATE — Conditional ATE
ATE conditional on covariates X. The "personalized" effect — same drug, different effect per patient.
Counterfactual
Y(1) for a unit that actually received T=0. Unobservable; estimated by the causal model.
§ 5.6.b — Method catalog (1/3)

Method catalog — RCT, PSM, IPW

Gold standard
RCT
Randomized Controlled Trial. Random assignment removes confounding by construction. Use whenever you CAN randomize.
Observational
Propensity Score Matching (PSM)
Match each treated unit with a similar untreated unit (similar P(T=1|X)). Assumes conditional ignorability — no unobserved confounder.
Observational
Inverse Propensity Weighting (IPW)
Reweight samples by 1/P(T|X). Same assumption as PSM, plus needs good overlap (no unit deterministically in one arm).
§ 5.6.b — Method catalog (2/3)

Method catalog — Doubly Robust, IV, RD

Belt + suspenders
Doubly Robust (AIPW, TMLE)
Combines outcome model + propensity model. Consistent if EITHER one is correctly specified — much safer than either alone.
Unobserved confounders
Instrumental Variable (IV)
Need a valid instrument: exogenous, affects Y only via T. Rare but powerful when found.
Threshold-assigned
Regression Discontinuity (RD)
Treatment given when a covariate crosses a cutoff (age, score). Compare units just above vs just below.
§ 5.6.b — Method catalog (3/3)

Method catalog — DiD, Synthetic Control, ML estimators

Before / after × treatment / control
Difference-in-Differences (DiD)
Compare change in Y over time between treated and control groups. Assumes parallel trends in the absence of treatment.
1 treated unit, many donors

Build a weighted combination of donor units that reproduces the treated unit's pre-treatment trajectory. Project counterfactual.
Heterogeneous CATE
Causal Forests / DML
Flexible ML nuisance models with theoretical guarantees. econml.CausalForest, dowhy.

§ 5.6.d — 4 steps explained

The four steps in detail

  • 1. Draw the DAG with domain experts — nodes = variables, arrows = assumed direct causal links. Version-control it next to the model code.
  • 2. Identify — does the DAG let you identify the effect? DoWhy checks back-door, front-door, and IV strategies automatically.
  • 3. Estimate — fit the chosen estimator (propensity matching, DML, etc.). Report point estimate + 95 % confidence interval.
  • 4. Refute — mandatory: (a) placebo outcome → expect effect ≈ 0, (b) random common cause → expect unchanged, (c) data subset → expect stable, (d) unobserved-confounder sensitivity (how strong would a hidden confounder need to be to null the result?).
  • An un-refuted causal estimate is a fragile claim. Skip step 4 and you ship a placebo.
§ 5.6.e — Worked example

Did the PID re-tuning reduce wear? — DoWhy

import pandas as pd
from dowhy import CausalModel

df = pd.DataFrame({
    "treatment":   ...,    # 1 if PID re-tuned, 0 otherwise
    "wear_index":  ...,    # continuous outcome
    "load":        ...,    # confounder
    "material":    ...,    # confounder
    "spindle_age": ...,    # confounder
})
model = CausalModel(
    data=df, treatment="treatment", outcome="wear_index",
    common_causes=["load", "material", "spindle_age"],
)
identified = model.identify_effect(proceed_when_unidentifiable=False)
estimate   = model.estimate_effect(identified,
                method_name="backdoor.propensity_score_matching")
print(estimate.value)        # e.g. -0.08 → tuning reduces wear by 0.08
refute = model.refute_estimate(identified, estimate,
                method_name="placebo_treatment_refuter")
print(refute)                # placebo effect should be ≈ 0
§ 5.6.f — Anti-patterns

Causal inference — five classic mistakes AVOID

  • Controlling for a mediator → closes the causal path, under-estimates the effect.
  • Controlling for a collider → opens a non-causal path, creates a spurious effect.
  • Reporting a regression coefficient as "causal effect" without a DAG — classic confound.
  • Treating self-selection as random assignment — propensity adjustment + sensitivity analysis are mandatory.
  • No refutation step — un-refuted causal claims do not survive contact with reality.
§ 5.6.g — Production gates

Gates before acting on a causal estimate MANDATORY

  • DAG signed off by a domain expert and version-controlled in the repo.
  • Overlap / positivity check: propensity scores in (ε, 1 − ε) — no unit near-certain to be in one arm.
  • All four DoWhy refutations pass: placebo, random common cause, data subset, unobserved confounder sensitivity.
  • Sensitivity analysis quantifies how strong a hidden confounder would have to be to null the result. Report the result.
  • If any of the four fail — do not act on the estimate. Investigate.

§ 6.3 — K-Means details

K-Means — the gotchas

  • Standardize first. K-Means uses Euclidean distance — features at different scales dominate.
  • Always run n_init=10 (default in modern sklearn) to escape local minima — see §4.14.
  • Find k via the elbow on inertia or via silhouette analysis (§6.4).
  • K-Means assumes spherical, equal-variance clusters. Anisotropic data → use Gaussian Mixture Model with full covariance instead.
  • K-Means cannot model noise / outliers — every point gets a label. Use DBSCAN if you have outliers.
  • For very large data: MiniBatchKMeans trades a small accuracy hit for a 10× speedup.

§ 6.3 — DBSCAN details

DBSCAN — choosing eps and min_samples

  • Pick min_samples ≥ 2 × n_features as a starting point.
  • Pick eps via the k-distance plot: sort all points by distance to their k-th neighbor, pick the elbow.

  • Never report "the clusters are clearly separated in the t-SNE plot" — t-SNE distorts global geometry.

§ 7.0 — Feedforward

ANN / MLP — feedforward baseline

  • Input modality: tabular features.
  • Typical use: simple regression / classification, RL policy head.
  • Key knobs: depth, width, dropout, activation.
  • Default baseline for tabular when boosted trees under-perform — rare but happens.
  • Needs feature scaling. Without standardization, training is unstable.
  • Almost never wins against XGBoost on tabular < 100k rows.
§ 7.0 — CNN 2D

CNN 2D — for images

  • Input modality: images (H × W × C).
  • Typical use: classification, detection, segmentation.
  • Key knobs: kernel size (3 or 5), stride, padding, channel progression.
  • Use a pre-trained backbone (ResNet, EfficientNet, ViT) + custom head — see §7.3.
  • Overkill for tabular data. Forcing tabular into 2D image breaks inductive bias.
  • For signals: prefer 1D CNN or TCN — covered next.
§ 7.0 — CNN 1D

CNN 1D — for signals

  • Input modality: 1D signals — vibration, audio, ECG.
  • Typical use: feature extraction + classification on fixed-length windows.
  • Key knobs: kernel along time axis (3, 5, 7), stride, dilation.
  • Strong baseline for vibration data. Cheaper to train than RNN, parallelizable.
  • Use BatchNorm between conv layers for stability.
  • Pair with global average pooling at the end — trivializes variable input length.

§ 7.0 — TCN

TCN — Temporal Convolutional Network

  • Input modality: long sequences (hundreds to thousands of steps).
  • Typical use: forecasting, RUL (remaining useful life), classification.
  • Key knobs: dilation factor (exponential), receptive field, residual blocks.
  • Outperforms LSTM on many industrial tasks. Parallelizable training, stable gradients, deterministic receptive field.
  • Default modern choice over LSTM for sequence length > 200.
  • Reference reading: Bai, Kolter & Koltun (2018) — "An Empirical Evaluation of Generic Convolutional and Recurrent Networks for Sequence Modeling".
§ 7.0 — RNN vanilla

RNN (vanilla) — educational, rarely production LEGACY

  • Input modality: short sequences.
  • Typical use: educational baseline only.
  • Key knobs: hidden size, sequence length.
  • Vanishing gradients beyond ~20 steps — limited representational power on real sequences.
  • Do not use vanilla RNN in 2026 for sequences > 20 steps. Keep as reference / pedagogy only.
  • Use LSTM, GRU, TCN, or Transformer instead — all covered next.
§ 7.0 — LSTM

LSTM — long short-term memory

  • Input modality: medium sequences (10–500 steps).
  • Typical use: language modeling, RUL, anomaly detection.
  • Key knobs: hidden size, number of layers, bidirectional flag, dropout.
  • Stable training; well-understood; mature ecosystem.
  • Slower than TCN. Use bidirectional only when the FULL window is available at inference time (not for streaming).
  • Use as fallback when TCN under-performs on your specific dataset.
§ 7.0 — GRU

GRU — gated recurrent unit

  • Input modality: same as LSTM, lighter.
  • Typical use: same as LSTM, edge deployment.
  • Key knobs: hidden size, number of layers.
  • ~25 % fewer parameters than LSTM with similar accuracy on most benchmarks.
  • Prefer GRU when memory or latency are constrained (edge MCU, mobile, real-time inference).
  • Both LSTM and GRU lose to TCN on long sequences — switch to TCN if seq_len > 500.
§ 7.0 — Transformer encoder

Transformer encoder — attention is all you need

  • Input modality: long sequences, text, signals (1k–10k tokens).
  • Typical use: classification, forecasting.
  • Key knobs: heads, layers, d_model, positional encoding scheme.
  • O(N²) memory in sequence length — explodes above ~2k tokens.
  • Use sparse / linear attention variants (Longformer, Reformer, Performer) for sequences > 2k.
  • Needs a large dataset (≥ 100k samples) OR transfer learning to avoid overfitting.
§ 7.0 — Transformer decoder

Transformer decoder / encoder-decoder — generation

  • Input modality: generation, translation, prognostics seq2seq.
  • Typical use: output is a sequence (not a label).
  • Key knobs: heads, layers, d_model + causal mask.
  • Needs large datasets — transfer learning is usually mandatory in industrial setups.
  • Use HuggingFace pretrained checkpoints + fine-tuning instead of training from scratch.
  • For prognostic seq2seq: consider Informer / TimesNet / TFT — TS-specialized variants.
§ 7.0 — ViT

Vision Transformer (ViT) — images

  • Input modality: images split into patches.
  • Typical use: classification.
  • Key knobs: patch size (16 typical), d_model, depth.
  • Needs ≥ 100k images OR pretraining to outperform CNNs.
  • For small image datasets: use a pre-trained ViT-Base from timm or HuggingFace and fine-tune.
  • Consider Swin Transformer for tasks needing locality (detection, segmentation).
§ 7.0 — Autoencoder

Autoencoder — reconstruction-based anomaly

  • Input modality: any (vector, image, sequence).
  • Typical use: reconstruction-based anomaly detection.
  • Key knobs: latent dimension, loss (MSE / cosine / contrastive).
  • Train on healthy data only; flag samples with reconstruction error above the p95 of healthy.
  • Use a held-out healthy validation set to set the threshold.
  • If healthy data has multiple modes, pure AE will smear. Use VAE or Mixture of AEs.
§ 7.0 — VAE

Variational Autoencoder (VAE)

  • Input modality: any.
  • Typical use: probabilistic generation, probabilistic anomaly detection.
  • Key knobs: latent dimension, KL weight β.
  • KL term regularizes the latent space — useful when you need likelihoods or smooth interpolation.
  • Beta-VAE (β > 1) encourages disentangled latents — feature axes become interpretable.
  • Reconstruction quality is typically lower than a plain AE — trade-off between smoothness and fidelity.
§ 7.0 — Conv AE

Convolutional Autoencoder

  • Input modality: images, signals (using 1D conv).
  • Typical use: denoising, anomaly detection on spatial / temporal data.
  • Key knobs: depth, bottleneck size.
  • Default architecture for vibration windows. 1D conv encoder + 1D deconv decoder, latent dim < 32.
  • Use skip connections (U-Net style) when reconstruction quality matters more than compression.
  • Pair with a Mahalanobis distance in latent space for tighter anomaly scoring.
§ 7.0 — GAN

GAN (vanilla / DCGAN / WGAN-GP)

  • Input modality: images, signals.
  • Typical use: synthetic data generation, augmentation of rare-class faults.
  • Key knobs: generator/discriminator balance, gradient penalty (WGAN-GP).
  • Notoriously unstable. WGAN-GP much safer than vanilla / DCGAN.
  • Useful to augment rare-class fault data when collecting more real samples is impossible.
  • Always track a diversity metric (Inception Score, FID) — generator-loss-only training masks mode collapse.
§ 7.0 — Diffusion

Diffusion (DDPM, score-based)

  • Input modality: images, signals.
  • Typical use: high-fidelity synthetic data generation.
  • Key knobs: number of timesteps (50–1000), noise schedule.
  • SOTA generation quality, much more stable training than GANs.
  • Heavy compute — both training and sampling. Consider only when GAN output is inadequate.
  • Pre-trained Stable Diffusion is a starting point for image work; for signals, see DiffWave, DiffSynth.
§ 7.0 — Graph nets

GCN / GraphSAGE / GAT

  • Input modality: graphs (nodes + edges + topology).
  • Typical use: fault propagation across machine fleets, sensor networks, supply chains.
  • Key knobs: message-passing layers, aggregation function (mean/sum/max/attention).
  • Use when the system IS a graph: plant layout, network of sensors, supply chain.
  • Overkill for a single machine — the graph collapses to a star, no useful inductive bias.
  • Library: PyTorch Geometric (torch_geometric) or DGL.
§ 7.0 — Contrastive SSL

Self-supervised contrastive (SimCLR, BYOL, MoCo)

  • Input modality: images, signals.
  • Typical use: pretraining on UNLABELED data; fine-tune on small labeled set.
  • Key knobs: augmentation pairs, temperature.
  • Critical when labels are scarce. Pretrain on years of unlabeled sensor history, fine-tune on the few labeled fault windows.
  • Default for vision: SimCLR or BYOL with strong augmentation pipeline.
  • For signals: pair with time-series-specific augmentations (jittering, time warping, channel masking).
§ 7.0 — Masked SSL

Masked modeling (MAE, BERT-style)

  • Input modality: signals, images.
  • Typical use: pretraining on unlabeled data — alternative to contrastive.
  • Key knobs: mask ratio (50–75 % typical for MAE).
  • Often simpler to tune than contrastive — single hyperparameter (mask ratio) vs many.
  • For sequences: mask random patches and predict them — same recipe as BERT.
  • For images: see Masked Autoencoder (He et al. 2021) — reference architecture.
§ 7.0 — Selection rules

Selection rules of thumb REFERENCE

  • Tabular, few features → XGBoost first (§4); deep learning only if it beats it on your data.
  • Vibration / audio 1D → 1D CNN or TCN before LSTM.
  • Sequences > 1000 steps → TCN or sparse Transformer, never vanilla LSTM.
  • Anomaly detection on healthy-only data → Autoencoder with reconstruction-error threshold.
  • Rare-class faults with < 100 examples → data augmentation first (§7.4); if insufficient, WGAN-GP / diffusion.
  • Labels scarce, unlabeled data abundant → self-supervised pretraining → fine-tune.
§ 7.0 — Anti-patterns

Deep learning — five anti-patterns

  • Vanilla RNN in 2026 for sequences > 20 steps — vanishing gradients, no business case.
  • Transformer trained on 5 k samples — will memorize, not generalize.
  • GAN trained with no diversity metric (only generator loss) — mode collapse invisible until production.
  • Mixing synthetic and real training data without labeling — model overfits to synthetic distribution.
  • "Let's just throw deep learning at it" without baseline comparison — see §4.1, run XGBoost first.

§ 7.1 — Chrono split

Chronological split — the golden rule

Time series have a direction. Random shuffling allows the model to peek at the future during training — guaranteed leakage, deceptively high test scores, catastrophic production failure.

from sklearn.model_selection import TimeSeriesSplit

tscv = TimeSeriesSplit(n_splits=5)
for fold, (tr, va) in enumerate(tscv.split(X)):
    # tr always precedes va in time
    assert X.index[tr].max() < X.index[va].min()
    model.fit(X.iloc[tr], y[tr])
    score = model.score(X.iloc[va], y[va])
    print(f"fold {fold}: score = {score:.3f}")

The earliest fold is small (CV grows the train set forward in time). Report the average across folds, but inspect each fold individually — degradation over time means concept drift.

§ 7.1 — Windowing

Windowing strategy

  • Window size W — covers at least one full cycle of the slowest dynamic you care about.
  • Step S — overlap = W − S. 50 % overlap (S = W/2) is a safe default.
  • Use numpy.lib.stride_tricks.sliding_window_view for zero-copy windowing — much faster than Python loops.
  • If labels change inside a window: assign label by majority vote AND filter out windows where the label transition occurs (boundary noise).
  • Per-window features go into a tabular dataset → train tree-based model. Per-window raw signal goes into a sequential model (CNN 1D, TCN, LSTM).

§ 7.1 — FFT features

FFT features per window

  • Peak frequency — argmax of the magnitude spectrum.
  • Peak magnitude — value at the peak frequency.
  • Energy in N bands — sum of |FFT|² across user-defined frequency ranges (e.g. 0–100 Hz, 100–500 Hz, etc.).
  • Spectral centroid — frequency-weighted mean (gravity center of the spectrum).
  • Spectral rolloff — frequency below which 85 % of the energy lies.
  • Always exclude the DC bin (index 0) — it captures the mean, often dominates and masks meaningful frequency content.

§ 7.2 — Pruning, PTQ, QAT

Pruning & quantization (PTQ / QAT)

Sparsify
Pruning
Zero out unimportant connections (magnitude-based or learned masks). Compressed size, sometimes faster — but only on hardware that exploits sparsity.
Post-training
PTQ — Post-Training Quantization
Reduce weight precision (FP32 → INT8) AFTER training. Fast, no retraining. Acceptable accuracy drop (1–3 %).
Train-aware
QAT — Quantization-Aware Training
Train with quantization simulators in the forward pass. Slower, but recovers most of the accuracy lost by PTQ.
§ 7.2 — Dynamic / Static / Low-rank

Dynamic quant, Static quant, Low-rank approximation

Runtime activations
Dynamic quantization
Weights pre-quantized; activations quantized on the fly per inference call. Best for Transformers and RNNs.
Calibrated
Static quantization
Both weights AND activations quantized using a representative calibration dataset. Faster than dynamic at inference. Best for CNNs.
Matrix factorization
Low-rank approximation
Replace a large W with U·V (rank-r factorization). Cuts parameters by orders of magnitude in fully-connected layers.
§ 7.2 — Binary / Distillation / Winograd / CUDA

Binary nets, Distillation, Winograd, CUDA

Ultra-low power
Binary networks
Weights ∈ {+1, −1}. Massive compression, large accuracy drop. Use only for ultra-constrained edge MCUs.
Big teacher → small student
Knowledge distillation
Train a small student to imitate a large teacher's soft outputs. Often recovers 90 %+ of teacher accuracy with 10× fewer params.
Conv acceleration
Winograd
Algebraic transform that reduces multiplications in 3×3 convolutions. Built into modern CUDA / cuDNN — typically transparent.
Training acceleration
Parallel GPU (CUDA)
Distribute across GPUs (Data Parallel, DDP, FSDP). Mandatory once training takes > 4 hours on single GPU.
§ 7.2 — Metrics

Compression metrics — what to log

  • mAP@0.5 — accuracy retention after compression (object detection benchmark — use F1 / accuracy for other tasks).
  • Model size (MB) — on-disk size of the saved model.
  • Latency p95 (ms) — inference time at the 95th percentile, measured on the target hardware (NOT on the dev laptop).
  • Sparsity (%) — fraction of zeroed weights (only meaningful with sparsity-aware hardware).
  • Compression ratio — original size / compressed size. Report alongside accuracy retention.
  • All five logged to MLflow per training run; persisted as model card fields.
§ 7.3 — Transfer learning

Transfer learning — fine-tune, don't restart

A model pre-trained on a large source dataset (ImageNet, BookCorpus) has already learned general features. Adapting the last layers to your task is much faster and more accurate than training from scratch with limited data.

Small data, similar domain
Feature extraction
Freeze ALL layers except a new head. Train only the head. Fastest, lowest risk of overfitting.
Medium data, different domain
Fine-tuning
Freeze first N layers (general features), train the last layers + head. Lower LR than from-scratch.
Large data, enough compute
Full fine-tuning
Unfreeze everything, train all layers. Best result when you can afford it.

§ 7.6 — Production gates

Time-series production gates MANDATORY

  • Backtest on a rolling-origin window (TimeSeriesSplit), never on a random fold.
  • Report p10, p50, p90 (probabilistic forecasts) — point forecast alone is insufficient for maintenance decisions.
  • Forecast SLA documented in the model card: horizon (h hours / d days), refresh cadence, p95 inference latency.
  • Check residual autocorrelation (Ljung-Box test) — un-modeled structure means the forecast can be improved.
  • Never use cross_val_score with default cv=KFold — it shuffles the time series and leaks the future.
§ 7 — Section recap

§ 7 Time Series & DL — take-aways

  • Pick architecture per modality (TCN for long signals, Conv1D for short, Transformer for huge text/seq).
  • TS preprocessing = chronological split + windowing + FFT + stationarity check. No shortcuts.
  • Compress for edge: PTQ first, QAT if accuracy drop is unacceptable, distillation for last-mile.
  • Transfer learning before training from scratch. Match strategy to dataset size + domain similarity.
  • For TS: tsfresh + XGBoost baseline first; jump to NeuralForecast only when justified.
  • Random k-fold on time series = guaranteed leakage. Use TimeSeriesSplit always.
